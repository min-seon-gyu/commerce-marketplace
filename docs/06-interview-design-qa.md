# 06. 면접 대비 — 설계 근거 심층 문답 (Why & Alternatives)

이 문서는 프로젝트의 주요 설계 결정마다 면접에서 받을 수 있는 두 갈래 질문에 답한다.

1. **왜 그렇게 설계했는가** (맥락·제약 → 결정)
2. **다른 방식은 선택지가 없었는가** (고려한 대안 → 왜 기각했나 → 트레이드오프)

각 항목은 `예상 질문 / 30초 답변 / 왜 / 고려한 대안 / 트레이드오프·한계 / 꼬리질문 대비`로 구성된다.

---

## 답변 프레임 (2단 질문 공통 전략)

면접관이 "왜 이렇게 했어요?" → "다른 방법은요?"로 파고들 때, 다음 4박자로 답하면 설득력이 산다.

1. **맥락(제약)**: "이 지점의 핵심 제약은 X였습니다" — 무엇을 지켜야 했는지(정합성/성능/유실 방지 등)를 먼저 못 박는다.
2. **결정**: "그래서 Y를 선택했습니다" — 한 문장으로.
3. **대안 비교**: "A도 검토했지만 ~때문에, B는 ~때문에 제외했습니다" — **대안을 알고 있고, 근거로 기각했음**을 보여준다. 이게 시니어리티의 신호다.
4. **트레이드오프 인정**: "대신 이런 비용을 감수했고, 이렇게 보완했습니다" — 완벽하다고 우기지 말고 약점과 보완책을 같이 말한다.

> 핵심: 면접관은 "정답"이 아니라 **"판단의 근거와 대안 인식"**을 본다. "그게 정석이라서"는 최악의 답이다.

## 1. 복식부기 원장 (Double-Entry Ledger)

**예상 질문**: "포인트/정산/결제 잔액을 그냥 컬럼으로 두면 되는데 왜 복식부기까지 도입했나요? 오버엔지니어링 아닌가요?"

**30초 답변**: 커머스의 돈은 결제·정산·포인트·프로모션 출연이 얽혀 있어서, 사고가 나면 "어디서 와서 어디로 갔는지"를 재구성할 수 있어야 합니다. 단순 잔액 컬럼은 그 이력을 못 남기고, 무엇보다 **전역 정합성을 검증할 수 없습니다**. 복식부기는 모든 돈의 이동을 차변/대변 2행으로 남겨, `Σ차변 = Σ대변`이라는 불변식으로 반쪽 분개(정합성 깨짐)를 배치가 자동 탐지합니다.

**왜 이렇게 설계했나**
- 핵심 제약은 **감사 가능성 + 자동 정합성 검증**. 돈을 다루는 시스템에서 "잔액이 왜 이 숫자인가"를 증명하지 못하면 장애 시 복구·대사가 불가능하다.
- `LedgerEntry`는 `@Immutable` + `amount > 0` 불변식. 되돌릴 땐 삭제가 아니라 **역분개(보상)**로만 → 원장이 "왜 금액이 바뀌었는가"를 스스로 증명한다.
- 잔액 캐시(`PointAccount.balance`)는 성능용 편의일 뿐, 진실의 원천은 항상 원장. 검증 배치가 `POINT_BALANCE` 원장 net == Σ(캐시 잔액)을 매일 대조한다.

**고려한 대안**

| 대안 | 장점 | 왜 기각했나 |
|------|------|-------------|
| **단순 잔액 필드** (`balance -= x`) | 가장 단순·빠름 | 이력 없음, 자금 추적 불가, 전역 불변식 없음 → 버그가 조용히 잔액을 오염시켜도 탐지 못 함 |
| **단식 거래 로그 테이블** (한 행에 +/- 금액) | 이력은 남음 | "돈 총합 보존" 불변식이 없어, credit 없이 debit만 기록되는 반쪽 버그를 검증할 수 없음 |
| **풀 이벤트 소싱(Event Sourcing)** | 완전한 재생·감사 | 스냅샷·프로젝션 재구축 등 인프라가 무겁다. 원장은 "돈에 한정한 append-only 이벤트 소싱"으로, ES의 감사 이득 대부분을 훨씬 가벼운 복잡도로 얻는다 |

**트레이드오프·한계**
- 쓰기 행이 2배(모든 거래가 최소 2행). → `LedgerService.record`에 항상 2행 저장을 **중앙화**해서 호출부가 실수할 여지를 없앴다.
- 0원 leg 문제(100% 할인 시 현금 leg가 0): `amount > 0` 불변식과 충돌 → **호출부에서 0원 leg를 생략**하고, 그래도 `SELLER_PAYABLE`은 다른 쌍이 gross를 채우도록 설계.

**꼬리질문 대비**
- *"정합성이 깨지면 자동 복구하나요?"* → 아니요. 검증 배치는 불일치 시 **자동 수정하지 않고** ERROR 로그 + 메트릭만 남깁니다. 돈은 사람이 판단해야 하고, 자동 보정은 원인을 숨겨 더 큰 사고를 만듭니다.
- *"왜 불변인가요?"* → 수정/삭제를 허용하면 감사 추적이 깨집니다. 역분개로만 되돌려야 "원 거래 + 취소 거래"가 원장에 모두 남습니다.
- *"판매자별 정산은 원장으로 어떻게?"* → 원장은 총액 기준이고, 판매자 귀속은 `OrderLine.sellerId`(비정규화)로 추적합니다(11번 참고).

## 2. 재고 동시성 — 분산락 + DB 비관적 락 이중 방어

**예상 질문**: "재고 초과판매를 어떻게 막았나요? 분산락이랑 DB 락을 둘 다 쓰는데, 하나로는 안 되나요?"

**30초 답변**: 두 락의 역할이 다릅니다. **Redisson 분산락은 성능(경합 완화)**, **DB 비관적 락(`SELECT ... FOR UPDATE`)은 정합성 최종 방어**입니다. 분산락은 정합성을 보장하지 못하고(락 TTL 만료·Redis 장애 시 두 스레드가 동시 진입 가능), DB 비관적 락만 쓰면 핫 SKU에 모든 요청이 DB 행 락으로 몰려 커넥션이 고갈됩니다. 그래서 **분산락으로 1차 직렬화해 DB 부하를 줄이고, DB 락으로 초과판매를 최종 차단**합니다.

**왜 이렇게 설계했나**
- 핵심 제약: **`quantity >= 0` 절대 불변(초과판매 0)** + 핫 SKU에서의 처리량.
- 순서가 정합성의 핵심: **락 획득 → `TransactionTemplate`로 트랜잭션 시작 → `SELECT FOR UPDATE` → 검증·차감 → 커밋 → 락 해제**. 락을 커밋 전에 풀면 다른 스레드가 커밋되지 않은 재고를 읽는다. 코드에서 `transactionTemplate.execute` 반환(커밋 완료) 후에 `withLock`의 `finally`가 unlock → 순서 보장.
- **재고 차감을 주문과 같은 트랜잭션에** 둔다(`deductWithinTx`는 `Propagation.MANDATORY`). 이후 단계(원장/포인트)가 실패하면 재고도 원자적으로 롤백 → 별도 보상 불필요.
- **데드락 방지**: 다품목 주문은 여러 SKU 락을 잡으므로 `skuId` 오름차순 **정준 락 순서**(`distinct().sorted()` 중첩 획득)로 교차 데드락을 막는다.
- **Redis 장애 대비**: 분산락 획득이 실패하면 예외를 삼키고 DB 비관적 락만으로 진행 → 정합성은 유지(분산락은 어디까지나 경합 완화용).

**고려한 대안**

| 대안 | 장점 | 왜 기각했나 |
|------|------|-------------|
| **DB 비관적 락 단독** | 정합성은 보장 | 핫 SKU에서 모든 요청이 같은 행 락에 직렬로 대기 → 커넥션 풀 고갈·lock wait timeout. 분산락이 DB 앞단에서 경합을 흡수해야 함 |
| **분산락 단독** | Redis만으로 빠름 | 분산락은 정합성 보장 수단이 아님(Redlock 논쟁). 락 TTL(10s) 만료·Redis 장애 시 두 스레드 동시 진입 → 초과판매. 정합성 앵커가 없음 |
| **낙관적 락(`@Version`)** | 락 대기 없음, 저경합에 유리 | 핫 SKU는 고경합 → 충돌·재시도가 폭증해 처리량이 오히려 나빠짐. 재고는 대표적 고경합 자원 |
| **원자적 조건부 UPDATE** (`UPDATE stock SET qty=qty-? WHERE qty>=?`) | 단일 쿼리로 원자 차감, 락 코드 불필요 | (아래 상세) |
| **Redis 재고 카운터(DECR)** | 매우 빠름 | 재고 진실원천이 Redis로 감 → 내구성/이중쓰기 문제, DB와 재조정 부담. 돈에 준하는 데이터라 DB가 원천이어야 함 |
| **큐 기반 SKU 단일 소비자**(파티션=skuId) | 락 자체가 불필요 | 비동기 체크아웃 UX·지연·운영 복잡도. 현재 규모엔 과함 |

**"왜 조건부 UPDATE를 안 썼나"** (실제로 이 프로젝트는 **read-then-write** 방식: `findBySkuIdForUpdate`로 행 잠금 → 엔티티 `deduct()`로 인메모리 차감 → dirty checking UPDATE. `Stock`에 `@Version`도 없어 순수 락 기반)
- 조건부 UPDATE는 훌륭한 대안이고 단건 차감엔 더 간결하다. 다만 (a) 차감은 **주문·원장·포인트를 아우르는 더 큰 트랜잭션의 일부**여야 하고, 도메인 규칙(`OUT_OF_STOCK` 예외를 정확한 시점에 던지기)을 태워야 해서 read-validate-act가 자연스럽다. (b) 조건부 UPDATE 단독은 핫 행의 DB 경합을 줄여주지 못한다 — 분산락의 역할(DB 앞단 경합 흡수)은 여전히 필요하다. 즉 조건부 UPDATE는 "정합성" 축의 대안일 뿐 "성능" 축을 대체하지 못한다.
- **정직한 인정 포인트**: read-then-write는 락에 전적으로 의존한다. 만약 분산락도 없고 비관적 락도 빠졌다면 lost update가 난다. 그래서 두 락 + `Propagation.MANDATORY`로 단독 커밋을 막는 방어가 필수다.

**트레이드오프·한계**
- Redis 의존·락 획득 지연이 추가된다 → Redis 장애 시 DB 락 fallback으로 정합성은 지키되 처리량은 떨어진다(가용성 > 처리량 선택).
- 락 leaseTime(10s) 안에 트랜잭션이 못 끝나면 락이 먼저 풀릴 수 있다 → 트랜잭션을 짧게 유지하고, 최종 정합성은 DB 락이 책임진다.

**꼬리질문 대비**
- *"검증은?"* → 재고 5인 SKU에 10건 동시 주문 → 정확히 5 성공/5 `OUT_OF_STOCK`, 최종 재고 0 (Testcontainers + `ExecutorService`/`CountDownLatch`로 실제 경합 재현).
- *"락 순서를 왜 정렬?"* → A가 sku1→sku2, B가 sku2→sku1을 잡으면 교차 데드락. 전역 정준 순서(오름차순)면 사이클이 생기지 않습니다.

## 3. 모듈러 모놀리스 vs 마이크로서비스

**예상 질문**: "요즘 다 MSA로 가는데 왜 모놀리스로 했나요? 나중에 못 쪼개면 어쩌죠?"

**30초 답변**: 이 시스템의 핵심 제약은 **결제 경로의 강한 정합성**입니다. 재고 차감·주문·원장·포인트·쿠폰 확정이 **한 DB 트랜잭션**에서 원자적으로 끝나야 합니다. MSA로 쪼개면 이 경로가 곧바로 분산 트랜잭션(saga + 보상)이 되어 돈의 정합성을 지키기가 훨씬 어려워집니다. 그래서 응집도는 모듈 경계로 얻고, 정합성은 단일 트랜잭션으로 얻는 **모듈러 모놀리스**를 택했습니다. 모듈 간 역방향 의존을 금지해 나중에 추출 경로도 열어뒀습니다.

**왜 이렇게 설계했나**
- 도메인별 모듈(`member`/`seller`/`product`/`inventory`/`cart`/`order`/`ledger`/`point`/`promotion`/`transaction`/`shipping`/`claim`)로 응집도를 높이되, 배포는 단일 앱으로 단순화.
- `order`가 재무·재고·배송을 오케스트레이션하고 **역방향 의존은 없음** → 의존 그래프가 단방향이라 나중에 특정 모듈을 서비스로 떼어내기 쉽다.
- 헥사고날 4계층(`domain`/`application`/`infrastructure`/`interfaces`)으로 의존은 항상 안쪽을 향한다.

**고려한 대안**

| 대안 | 장점 | 왜 기각했나 |
|------|------|-------------|
| **MSA** | 독립 배포/스케일, 팀 경계 | 결제 경로가 분산 트랜잭션(saga)으로 변함 → 재고·원장·포인트의 원자성을 보상으로 흉내내야 하고, 결과적 일관성이 도처에 퍼져 돈 정합성 난이도 급상승. 운영 오버헤드(서비스 디스커버리·메시 ·관측)도 큼 |
| **전통 레이어드 모놀리스**(기술 계층으로만 분리) | 익숙함 | 도메인이 뒤섞여 응집도가 낮고 커질수록 ball-of-mud. 모듈 경계가 없어 나중에 MSA 추출도 어려움 |

**트레이드오프·한계**
- 단일 장애점·모듈별 독립 스케일 불가·공유 DB. 현재 규모에선 수용 가능한 비용이고, 모듈 경계 + 무역방향 의존이 마이그레이션 경로를 열어둔다.

**꼬리질문 대비**
- *"쪼갠다면 뭐부터?"* → 결제 경로와 트랜잭션 결합이 약하고 읽기 부하가 큰 **`product`(카탈로그)**부터. 조회 전용이라 독립 스케일 이득이 크고 돈 정합성과 분리돼 있습니다.
- *"언제 쪼개나?"* → 특정 모듈이 독립 스케일이 필요하거나 팀 경계가 생길 때. 지금 쪼개는 건 얻는 것 없이 분산 트랜잭션 비용만 사는 것.

## 4. 이벤트 전달 — Transactional Outbox (vs AFTER_COMMIT / CDC / 2PC)

**예상 질문**: "주문 이벤트를 Kafka로 보내는데 왜 굳이 Outbox 테이블을 거치나요? 그냥 커밋 후에 발행하면 안 되나요?"

**30초 답변**: DB 커밋과 메시지 발행은 **이중 쓰기(dual write)** 문제입니다. 커밋 후 발행(`AFTER_COMMIT`)은 커밋과 발행 사이에 앱이 죽으면 **이벤트가 유실**됩니다. 그래서 이벤트를 주문과 **같은 트랜잭션(BEFORE_COMMIT)**에 outbox 행으로 원자 캡처하고, 릴레이가 폴링해 Kafka로 발행합니다. "주문 커밋 ⇔ 이벤트 캡처"가 원자적이라 유실이 0입니다.

**왜 이렇게 설계했나**
- 핵심 제약: **주문 후속 처리(이력·알림·분석)를 결제 tx에서 분리**하되 **유실은 0**.
- 원자 캡처: 이벤트를 주문 tx의 BEFORE_COMMIT에서 `order_outbox_events`에 기록.
- at-least-once + **멱등 소비**: 소비자는 `existsByEventId`로 중복 적용 무시.
- **DLT**: apply 실패는 지수 백오프 재시도 후 `order-events.DLT`로 적재(무성 유실 차단), DLT 전송 결과까지 검증.
- **재조정(reconciliation)**: "발행됐으나 미적용" 행을 anti-join으로 찾아 멱등 재적용.
- **poison 가드**: 영구 실패 행은 attempts 누적 후 격리(quarantine) → head-of-line 블로킹 차단.
- **킬스위치**(`order.kafka.enabled=false`): 브로커 없이 릴레이가 직접 적용 → 로컬/CI 부담 제거(12번 참고).

**고려한 대안**

| 대안 | 장점 | 왜 기각했나 |
|------|------|-------------|
| **`@TransactionalEventListener(AFTER_COMMIT)` 직접 발행** | 가장 단순 | 커밋과 발행 사이 크래시 시 이벤트 유실(dual write). 보장 전달 불가 |
| **BEFORE_COMMIT 직접 Kafka 발행** | 커밋 전에 보냄 | 커밋이 롤백돼도 이미 발행된 유령 이벤트 발생 + 브로커 지연이 커밋을 블로킹 |
| **2PC/XA (DB+Kafka 분산 트랜잭션)** | 원자성 이론상 보장 | Kafka의 XA 지원이 빈약, 2PC는 가용성·성능·코디네이터 복잡도 비용. 업계 합의는 "2PC보다 Outbox" |
| **CDC (Debezium, binlog 구독)** | 앱측 폴링 불필요, 진짜 디커플링, 이중쓰기 없음 | Kafka Connect+Debezium 인프라·binlog 접근이 무겁고, 변경 이벤트가 테이블 형태(도메인 이벤트 아님). 단일 앱 규모엔 self-contained한 outbox 폴링이 추론하기 쉬움 |

> CDC는 "대규모 프로덕션에서 이 outbox의 진화형"이라고 정직하게 인정하면 좋다 — 대안을 이해하고 있음을 보여준다.

**트레이드오프·한계**
- 폴링 지연·outbox 테이블 증가(정리 필요)·릴레이라는 움직이는 부품 추가. → 킬스위치·재조정·격리로 운영 리스크를 관리.

**꼬리질문 대비**
- *"왜 Kafka? RabbitMQ/SQS는?"* → 로그 보존 기반 재생(replay), 파티션 순서 보장, 다중 소비자 fan-out(알림·분석 추가) 때문. 같은 토픽에 소비자를 붙여 확장한다.
- *"중복 적용은?"* → at-least-once이므로 소비자가 `eventId` 멱등 체크로 흡수.

## 5. 멱등성 — Redis + DB 이중 저장 (vs 단독 / 409)

**예상 질문**: "더블클릭이나 재시도로 주문이 중복되는 걸 어떻게 막았나요? Redis랑 DB를 둘 다 쓰는 이유는요?"

**30초 답변**: `Idempotency-Key` 헤더로 "정확히 1회"를 보장합니다. **DB의 unique 제약이 정합성 앵커(분산 뮤텍스)**이고, **Redis는 빠른 감지 경로**입니다. Redis만 쓰면 eviction/장애 시 중복 주문이 뚫리고, DB만 쓰면 흔한 중복 케이스마다 DB를 왕복합니다. 그래서 Redis로 빠르게 걸러내고 DB unique로 최종 보장합니다.

**왜 이렇게 설계했나**
- 동시 요청은 `reserve()`가 IN_PROGRESS 행을 INSERT → **unique 위반이면 하나만 선점 성공**, 나머지는 걸러진다.
- 키는 **사용자 스코프**: `SHA-256(memberId + method+URI + rawKey)`로 축약. 다른 사용자/엔드포인트가 같은 `Idempotency-Key`를 보내도 충돌·응답 유출이 없다.
- 재시도 시 **원 응답 본문 + 원 상태코드(201)** 를 그대로 반환한다(`IdempotencyResponseAdvice`가 201을 캡처·재반환) → 중복을 409로 만들지 않아 클라이언트가 정상 흐름을 잇게 함.
- **exactly-once**: `afterCompletion`에서 비즈니스 커밋 성공 시 선점을 유지하고, 실패 시에만 release해 재시도를 허용.

**고려한 대안**

| 대안 | 장점 | 왜 기각했나 |
|------|------|-------------|
| **Redis 단독** | 가장 빠름 | eviction·장애 시 멱등 기록 소실 → 중복 주문. 돈엔 내구성 부족 |
| **DB unique 단독** | 내구성·단순 | 정합성엔 충분하나 흔한 중복도 매번 DB 왕복. Redis 빠른 경로가 상시 부하를 흡수 |
| **중복을 409 Conflict로 반환** | 구현 단순 | 클라이언트 재시도가 원 성공(201)을 못 받아 에러 처리를 강제당함. 재시도가 "정상 흐름을 잇는" UX를 위해 원 응답 재현을 택함 |

**트레이드오프·한계**
- 이중 저장 복잡도·TTL(Redis 24h) 관리·고아 IN_PROGRESS 정리 필요. → `IdempotencyCleanupScheduler`가 TTL 초과 IN_PROGRESS만 회수(COMPLETED는 보존)해 poison key를 막는다.

**꼬리질문 대비**
- *"in-progress 중복은?"* → 선점 성공 후 아직 처리 중인 동시 중복 요청엔 409를 반환합니다(이건 재시도가 아니라 진짜 동시 요청이므로).
- *"왜 키를 해싱?"* → 사용자·엔드포인트·원본 키를 합쳐 SHA-256(64 hex)으로 고정 폭 컬럼에 담고, 스코프 격리를 한 번에 얻기 위함.

## 6. 부분 환불 — 최대잔여 배분 + `@Version` 낙관락

**예상 질문**: "라인 단위 부분 환불에서 할인·포인트를 어떻게 정확히 나눴나요? 여러 번 나눠 환불하면 반올림 오차가 안 쌓이나요?"

**30초 답변**: 두 문제를 나눠 풀었습니다. (1) **금액 정확성**은 라인별 독립 반올림 대신 **최대잔여(largest-remainder) 배분**으로, 전체 라인을 기준으로 결정적으로 나눠 반복·누적 환불에도 배분 합이 원값과 정확히 일치합니다. (2) **동시성**은 `Order.refundedAmount` 누산기 갱신이 매번 `orders` 행에 versioned UPDATE를 유발하게 해서 **`@Version` 낙관락으로 직렬화**했습니다.

**왜 이렇게 설계했나**
- **최대잔여 배분**: 라인 금액 비율로 할인 D·적립 P를 나누되 floor 후 남은 잔여 단위(0.01/1P)를 잔차가 큰 라인부터 1단위씩 결정적으로 분배. **전체 라인 기준**이라 부분 환불을 반복·누적해도 합이 원 D·원 P와 정확히 일치(페니 드리프트 없음).
- **`@Version` 낙관락**: 누산값 변경 → versioned UPDATE → 동시 환불이 낙관락으로 직렬화되고 밀린 tx는 예외로 롤백. 상태(`PAID → PARTIALLY_REFUNDED → REFUNDED`)는 누적값이 결정 → 상태 고착·이중 반영 없음.
- **in-tx 이중환불 재확인**: 트랜잭션 안에서 대상 라인을 다시 로드해 `OrderLine.refunded`를 재검사 + 재고락 → 라인 단위 이중 환불 차단.

**고려한 대안**

| 대안 | 장점 | 왜 기각했나 |
|------|------|-------------|
| **라인별 독립 비례 반올림** | 구현 단순 | Σ(배분) ≠ 원값(페니 드리프트). 여러 번 나눠 환불하면 오차 누적 |
| **마지막 라인에 잔차 몰기** | 단순 | 환불 순서에 의존(비결정적)·불공정. 최대잔여는 표준 배분법(의석 배분 등)으로 결정적 |
| **주문에 비관적 락** | 동시 환불도 차단 | 행 락을 더 오래 점유. 한 주문의 동시 환불은 드물어 저경합 → 낙관락이 더 가벼움. 누산기 패턴이 자연스럽게 versioned UPDATE를 유발 |

**트레이드오프·한계**
- 경합에서 밀린 환불 tx는 예외로 롤백된다(재시도 안 함, 문서화된 단순화). 재무 손실은 없고, 운영자가 재시도/거절로 정리.

**꼬리질문 대비**
- *"전액 환불이면?"* → 모든 라인이 환불되면 배분 합이 원값과 일치 → 계정별 net 0으로 복귀(발송 전 전체 취소와 동일한 재무 결과).
- *"정산엔 어떻게 반영?"* → `sumSellerSalesInPeriod`이 `ol.refunded=true` 라인을 제외하고 `PAID`·`PARTIALLY_REFUNDED`의 잔여 라인만 합산 → 환불 매출이 정산에서 자동 제외.

## 7. 트랜잭션 경계 — 원장은 동기, 이벤트는 비동기 (경계 원칙)

**예상 질문**: "이벤트 드리븐(4번)을 쓰면서 왜 원장 기록은 이벤트로 안 하고 동기로 했나요? 기준이 뭔가요?"

**30초 답변**: 기준은 **"핵심 자금 정합성이냐, 부수효과냐"**입니다. 잔액 변경과 원장 분개는 **반드시 같은 DB 트랜잭션**이어야 합니다 — 이벤트로 미루면 커밋 후 장애 시 원장이 누락돼 정합성이 깨집니다. 반면 이력·알림·분석 같은 **비핵심 부수효과**는 커밋 후 유실돼도 재조정으로 복구 가능하니 Outbox로 비동기화합니다.

**왜 이렇게 설계했나**
- 결제 시 재고 차감·주문/라인·거래/원장·포인트 적립·배송(PREPARING) 생성·쿠폰 확정이 **모두 같은 트랜잭션** → 하나라도 실패하면 전부 롤백(원자성).
- 원장을 이벤트로 미루면 "잔액은 바뀌었는데 원장은 없는" 창이 생긴다 → 절대 불가.
- 조회 경로는 `@Transactional(readOnly=true)`.

**고려한 대안**
- **원장도 이벤트/Kafka로 비동기**: 결합은 낮아지지만 커밋 후 장애 시 원장 누락 → 정합성 붕괴. 돈의 원장은 부수효과가 아니라 **핵심 상태**라 기각.

**트레이드오프·한계**
- 결제 트랜잭션이 길어진다(여러 모듈을 한 커밋에). → 트랜잭션을 짧게 유지하고, 비핵심은 전부 Outbox로 밀어내 트랜잭션 범위를 최소화.

**꼬리질문 대비**
- *"그럼 Outbox 캡처도 tx 안이잖아요?"* → 네, outbox **쓰기**만 tx 안(원자 캡처)이고, Kafka **발행**은 릴레이가 tx 밖에서 합니다. 캡처와 발행을 분리한 게 outbox의 핵심.

## 8. 쿠폰 예산 — Redis-Lua 원자 카운터 + 트랜잭션 밖 reserve/release

**예상 질문**: "쿠폰 예산 초과를 어떻게 막나요? 왜 DB가 아니라 Redis-Lua로, 그것도 트랜잭션 밖에서 예약하나요?"

**30초 답변**: 예산은 핫한 공유 카운터라 DB 행 락으로 처리하면 결제 트랜잭션 안에서 경합이 병목이 됩니다. 그래서 **Redis-Lua로 원자적 조건부 증가**(한도 초과면 롤백)를 하고, 결제 **트랜잭션 밖에서 먼저 예약(reserve)**한 뒤 다운스트림(락/tx)이 실패하면 **release로 보상**합니다.

**왜 이렇게 설계했나**
- **Redis-Lua 원자성**: `INCRBY` 후 한도 초과면 `DECRBY`로 원자 롤백. 단순 `INCR`로는 "한도 이하일 때만 증가"라는 조건부 연산을 원자로 못 한다 → Lua로 check-and-increment를 한 왕복에.
- **tx 밖 reserve**: 예산 예약을 DB 트랜잭션 수명에 묶지 않아, 긴 tx나 롤백과 예산 카운터를 분리. 실패 시 명시적 `release`로 되돌린다.
- **release 하한 0 클램프**: `DECRBY` 후 음수면 0으로 SET → 카운터가 음수로 새는 것 방지.

**고려한 대안**
- **DB 예산 행 + 비관적 락/조건부 UPDATE** (`UPDATE promotion SET spent=spent+? WHERE spent+?<=limit`): 정합성은 되지만 핫 프로모션 행에 경합이 몰리고, 그 락을 결제 tx가 물고 있어 다른 락과 함께 오래 점유 → Redis로 오프로딩.

**트레이드오프·한계 (중요·정직하게)**
- reserve와 DB tx가 별개 자원이라, **reserve 후 release 전에 앱이 크래시하면 예산이 과예약된 채 샐 수 있다**. → **resync 스케줄러(묶음 G, 기본 5분)**가 DB 실집계(`sumConsumedBudgetByPromotion`)를 진실원천으로 Redis 카운터를 재동기화(멱등 `set`)해 드리프트를 자가 치유한다. DB 집계는 `CANCELLED` 주문만 제외하므로 취소=release / 환불=유지 정책과도 정합적.

**꼬리질문 대비**
- *"쿠폰 발급 중복은?"* → 발급도 `@Idempotent`라 재시도해도 한 번만 발급.
- *"reserve를 tx 안에 넣으면 안 되나?"* → 넣을 수도 있지만, 롤백 시 Redis 카운터를 되돌릴 훅이 애매하고(트랜잭션 동기화 콜백 필요), 핫 카운터를 tx 수명에 묶으면 tx가 길어질수록 예산 홀드가 길어진다. 밖에서 reserve+명시적 release가 통제하기 쉽다.

## 9. 워크플로우 무결성 — 배송 상태 게이트 + 반품 승인 트랜잭션 콜백

**예상 질문 A**: "이미 배송된 상품을 구매자가 셀프 취소하면 상품도 갖고 환불도 받는데, 어떻게 막았나요?"
**예상 질문 B**: "반품 승인 시 환불과 클레임 완료를 어떻게 원자적으로 처리했나요?"

**30초 답변 A**: `Shipment` 상태를 **게이트**로 씁니다. 취소는 발송 전(`PREPARING`)에만 허용하고, 발송/배송 후 환불은 셀프 취소가 아니라 **반품 클레임 → 운영자 승인** 경로로만 가능합니다. 구매자 직접 환불 엔드포인트(`POST /orders/{id}/refund`)는 아예 제거했습니다.

**30초 답변 B**: `approveReturn`에 `@Transactional`을 붙이지 않고, 환불(`refundLines`)이 여는 트랜잭션 안에서 **콜백**으로 클레임을 완료 처리합니다. `refundLines`는 이미 "분산락→tx 시작→커밋→락 해제" 순서를 소유하는데, 승인 서비스를 밖에서 `@Transactional`로 감싸면 tx가 락보다 바깥에서 열려 순서가 어긋나기 때문입니다.

**왜 이렇게 설계했나 (게이트)**
- `Shipment`(주문당 1건)의 `PREPARING → SHIPPED → DELIVERED` 상태 하나가 **취소(발송 전)와 반품(배송완료 후) 게이트를 동시에** 담당한다 — 게이트 진실원천이 하나.
- 인가 계층화: 발송/배송완료 전이·클레임 승인/거절은 `/admin/**`(ADMIN), 배송 조회·반품 요청은 인증 + 컨트롤러에서 본인 자원만.

**왜 이렇게 설계했나 (콜백 원자성)**
- `refundLines(memberId, orderId, lineIds, afterRefundInTx)`에 트랜잭션 콜백을 넘겨, 환불과 클레임 `COMPLETED` 전이를 같은 커밋에 묶는다 → 부분 성공(환불은 됐는데 클레임이 `REQUESTED`에 남는 상태)을 원천 차단.
- 락/tx 경계는 `refundLines`가 단독 소유하므로 "락 획득 → tx 커밋 → 락 해제" 순서가 보존된다.

**고려한 대안**

| 대안 | 왜 기각했나 |
|------|-------------|
| **order에 `canCancel` 플래그로 게이트** | 배송 상태와 별개의 진실원천이 생겨 드리프트 위험. `Shipment` 상태 하나로 통일 |
| **`approveReturn`에 `@Transactional`** | tx가 `refundLines`의 분산락보다 바깥에서 열려 "락 해제가 커밋보다 앞설 수 있음" → 순서 붕괴. 그래서 콜백 방식 |
| **구매자 직접 환불 엔드포인트 유지** | 배송 후 워크플로우 우회(상품 보유+전액 환불) 가능 → 엔드포인트 자체를 제거 |

**트레이드오프·한계**
- 콜백 방식은 관례적인 `@Transactional`보다 읽기 흐름이 덜 직관적이다(트랜잭션 경계가 호출된 메서드 안에 숨음) → 주석·문서로 의도를 명시.
- 같은 라인 동시 반품/동시 승인에서 밀린 쪽은 환불 안 되고 `REQUESTED`로 남는다(이중 환불은 없음) → 운영자가 재시도/거절로 정리(문서화된 단순화).

**꼬리질문 대비**
- *"콜백 없이 REQUIRES_NEW 쓰면?"* → 별도 tx가 되어 원자성이 깨집니다(환불 커밋 후 클레임 전이 실패 창). 같은 커밋이어야 해서 콜백.

## 10. 정산 — Spring Batch + 확정 시 재계산

**예상 질문**: "정산을 왜 배치로? 그리고 왜 확정 시점에 금액을 다시 계산하나요?"

**30초 답변**: 정산은 본질적으로 **주기적(일/주/월)**이고 판매자 전체를 안정적으로 처리해야 해서 Spring Batch로 청크·재시작·스킵을 얻습니다. 확정 시 **재계산**하는 이유는, calculate 이후 확정 전에 취소·환불이 들어오면 스냅샷을 믿을 경우 **과지급**이 나기 때문입니다.

**왜 이렇게 설계했나**
- `settlementJob = calculate → verify`. Reader(`JpaPagingItemReader`, APPROVED 판매자 페이징) → Processor(구간 정산 생성, 중복/0원 filter) → Writer(일괄 저장).
- `faultTolerant().skip(...)`으로 단건 실패 격리 후 완주, `RunIdIncrementer` + 프로세서 사전 중복 체크로 **재실행 멱등**.
- 매출 집계는 환불 라인 제외 + `PAID`·`PARTIALLY_REFUNDED` 잔여 라인만 → 부분 환불이 정산에 정확 반영.
- 확정(`confirm`) 시점 **재계산**으로 calculate 이후 취소·환불분 과지급 방지(스냅샷 신뢰 금지).
- 起動 자동 실행은 끄고(`spring.batch.job.enabled=false`) 스케줄러(03:00 KST) + 관리자 엔드포인트로 트리거, 메타 스키마는 Flyway 소유.

**고려한 대안**

| 대안 | 왜 기각했나 |
|------|-------------|
| **실시간 정산**(주문마다 즉시) | 쓰기 부하 큼, 환불/조정 반영이 어렵고, 정산은 애초에 기간 개념(비즈니스적으로 주기적) |
| **calculate 스냅샷 신뢰** | calculate~confirm 사이 환불 시 과지급 → 확정 시 재계산으로 방지 |
| **단순 루프 배치**(Batch 미사용) | 청크·재시작·skip·페이징을 직접 구현해야. 대량 판매자 처리·부분 실패 격리·재실행 멱등을 Batch가 제공 |

**꼬리질문 대비**
- *"중복 정산 방지는?"* → `(seller, periodStart, periodEnd)` 유니크 + 프로세서 사전 중복 체크.
- *"검증 스텝은 뭘?"* → 이상치(0 이하 정산)를 점검하고 메트릭을 남깁니다.

## 11. 쿠폰 부활 정책 — 취소=복원 / 환불=유지 (의도적 비대칭)

> ✅ **문서 정합성(해소 완료)**: 과거 `docs/01-domain-design.md`가 "취소·환불은 쿠폰을 되살리지 않는다"고 서술했으나 실제 코드는 **취소 시엔 복원**한다. 이 불일치는 doc 01(체크아웃 할인 항목)과 doc 03(취소·환불 분개 절)을 실제 동작으로 수정해 해소했다. 아래가 실제 동작이다.

**예상 질문**: "주문 취소하면 썼던 쿠폰이 되살아나나요? 환불은요? 왜 다르게 처리했나요?"

**30초 답변**: 의도적으로 **비대칭**입니다. **발송 전 전체 취소는 쿠폰을 복원(REDEEMED→ISSUED)하고 예산도 release**합니다 — 주문이 아예 이행되지 않았으니 쿠폰 혜택을 돌려주는 게 맞습니다. 반면 **배송완료 후 반품 환불은 쿠폰·예산을 소진 상태로 유지**합니다 — 단일 사용 쿠폰이 이미 이행된 주문에서 정당하게 소비됐으므로 환불로 되살리지 않습니다.

**왜 이렇게 설계했나**
- 각 흐름 **내부에서는 쿠폰 status와 예산이 함께** 움직인다: 취소는 둘 다 복원, 환불은 둘 다 유지. 두 흐름 **사이만** 비대칭.
- DB 진실원천과도 정합적: 예산 소비 집계(`sumConsumedBudgetByPromotion`)가 `CANCELLED`만 제외 → 취소 주문은 소비에서 빠지고(release와 일치), 환불 주문(`PARTIALLY_REFUNDED`/`REFUNDED`)은 소비로 계속 집계(유지와 일치) → resync 배치가 이 결정을 되돌리지 않음.

**고려한 대안**
- **항상 복원**: 환불(배송완료 후)에도 쿠폰을 되살리면 단일 사용 쿠폰이 부활해 재사용 가능 → 프로모션 남용. 기각.
- **항상 유지**: 발송 전 취소(미이행)에도 쿠폰을 안 돌려주면 고객에게 불리하고 예산도 잠긴 채 샌다. 취소는 돌려주는 게 합리적.

**꼬리질문 대비**
- *"환불인데 쿠폰을 안 돌려주면 고객 불만 아닌가요?"* → 배송완료까지 이행된 거래이고, 쿠폰은 그 구매에 이미 쓰였습니다. 필요하면 정책적으로 재발급으로 보상할 수 있고, 이건 "단일 사용 쿠폰의 부활 방지"라는 **의도적 단순화**입니다.

## 12. 심화 묶음 (짧게 — 꼬리질문용)

### 12-1. `OrderLine.sellerId` 비정규화
- **왜**: 판매자별 정산 합산을 라인만으로 처리하려고 `sellerId`를 라인에 복제. 정산 쿼리가 `product`까지 조인할 필요가 없어지고, 커버링 인덱스(`seller_id, refunded, order_id, line_amount`)로 인덱스만 읽어 정산 합산을 가속(×8.4).
- **대안**: 정규화 유지(조인) → 정산 배치의 대량 합산에서 힙 접근·조인 비용. 비정규화는 쓰기 시 상품↔라인 sellerId 일관성 책임을 지지만, 주문 시점 스냅샷이라 사후 변경도 안전.

### 12-2. Flyway 단일 진실 + `ddl-auto=validate`
- **왜**: 스키마를 Flyway가 소유하고 Hibernate는 검증만(`validate`) → 운영에서 자동 DDL로 인한 사고를 원천 차단, 마이그레이션 이력이 코드로 남음.
- **대안**: `ddl-auto=update`(자동 반영)는 편하지만 예측 불가한 스키마 변경·운영 위험. 기각.

### 12-3. JWT principal = `memberId`, 본문 memberId 불신
- **왜**: 컨트롤러가 `SecurityUtils.currentMemberId()`로 신원을 도출하고 **요청 본문의 memberId는 신뢰하지 않는다** → 타인 자원 접근·권한 상승 차단. 멱등 키도 사용자 스코프로 격리(5번).
- **대안**: 본문/파라미터 memberId 신뢰 → IDOR 취약점. 기각.

### 12-4. Kafka 킬스위치 (`order.kafka.enabled`) — dev/prod parity 트레이드오프
- **왜**: `false`면 릴레이가 직접 적용해 브로커 없이 동작 → 로컬/CI에서 Kafka 기동 없이 통합 테스트. Kafka 파이프라인 테스트만 `@EmbeddedKafka`로 오버라이드.
- **트레이드오프**: dev와 prod의 전달 경로가 달라지는 parity 손실. 대신 동일 outbox→apply 로직을 공유하고, Kafka 경로 자체는 별도 E2E 테스트로 검증해 리스크를 좁힘.

### 12-5. 성능 최적화의 "실증" 방법론
- **왜**: 수치를 지어내지 않고 통합 테스트로 실증. N+1은 실제 SQL 문 수를 세어 상수임을 단언, 인덱스는 **동일 데이터에 `FORCE/IGNORE INDEX`만 토글**해 인덱스 자체 효과만 격리 + `EXPLAIN`으로 실제 사용 컬럼 확인 + 결과값 일치도 단언("빠른데 틀린" 최적화 배제).
- **대안**: 벤치마크 수치만 제시 → 재현 불가·머신 의존. 그래서 시간은 로그로만, 강제 실패 단언은 결정적인 것(SELECT 수·`EXPLAIN`·결과 일치)에 둠.

## 한 장 요약 — 관통하는 설계 철학

이 프로젝트의 결정들을 하나로 꿰는 원칙:

1. **정합성은 타협하지 않는다** — 돈 경로는 단일 트랜잭션·복식부기·검증 배치로 강하게. (원장 동기, 모놀리스, `MANDATORY`)
2. **성능과 정합성을 다른 계층으로 분리한다** — 분산락(성능)+DB락(정합성), Redis 예산(성능)+DB 집계(진실원천).
3. **유실보다 중복을 택하고 멱등으로 흡수한다** — at-least-once + `eventId`/`Idempotency-Key` 멱등.
4. **되돌림은 삭제가 아니라 보상이다** — 역분개·보상 거래·release. 이력이 남아야 감사된다.
5. **한계를 숨기지 않고 자가 치유로 감싼다** — resync·재조정·cleanup 스케줄러.
